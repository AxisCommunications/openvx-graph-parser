Source Code for Package graph_parser.graphml_parser

  1  """Graph parser package for parsing OpenVX graphs saved in the yEd graphml format 
  2   
  3  Introduction 
  4  ============ 
  5   
  6  The yEd graph is expected to contain parser-specific node data for how to set up the 
  7  autogenerated OpenVX C-code that handles creating graph skeletons and switching I/O images. 
  8   
  9  In order for the parser to understand how to generate the graph source code correctly, 
 10  a few instructions need to be followed. 
 11   
 12  Where to add metadata information to nodes 
 13  ------------------------------------------ 
 14   
 15  To add parameters or other information to any nodes (function nodes and image nodes), 
 16  right click on the node, and select "Properties". Select the "Data" tab and enter the 
 17  needed data in the "Description" field. All additional information needed to process 
 18  nodes is added here except if settings that are global for the whole graph. 
 19  (see the UserData object for this exception). 
 20   
 21  Parameter information in the Data field is given within square brackets, 
 22  first specifying the type of information given (nodetype, vx_df_image_e etc.), and after space separation, 
 23  giving the value of that parameter (e.g. input_image[3] or VX_DF_IMAGE_U8, DYNAMIC etc.). 
 24   
 25  Note that if parameters have been added to the Data field of a node, 
 26  one can see them as a popup if one hovers with the mouse pointer 
 27  over that node in the yEd editor. If no parameters have been added, 
 28  the popup only displays the name of the node. 
 29   
 30  Adding a function node to the graph 
 31  ----------------------------------- 
 32   
 33    1. Function nodes should be flowchart objects of type "Start1" or "Start2" 
 34       (choose the one that fits the diagram best visually) 
 35    2. The function nodes should be named after the OpenVX function they represent, 
 36       but omitting the "vx" and the "Node" part of the node name. E.g. for the function 
 37       node named  "vxHalfScaleGaussianNode" in the standard, one should name the function 
 38       node "HalfScaleGaussianNode". The name is added to the text field of the node. 
 39       The reason to omit the "vx" and "Node" parts is to make names in the graph shorter and easier to read. 
 40    3. We also need to add (to the "Data" tab) any needed parameters as specified in the OpenVX standard. 
 41       They are added with square brackets in the same way as for image nodes. 
 42       For instance, for vxHalfScaleGaussion the OpenVX standard requires the parameter "vx_int32 kernel_size". 
 43       We follow that notation and add the information in the Data field of the function node: 
 44       [vx_int32 5], for a kernel size of 5 pixels. 
 45    4. By writing a I{dynamic_type} followed by a type name within square brackets, 
 46       (e.g. [dynamic_type vx_int32[index]]) the python parser interprets I{dynamic_type} as a directive to the parser 
 47       to treat the named type as a dynamic parameter whose value can be changed between graph executions. 
 48       Indexing should start from 0 and have no "gaps". 
 49       One can still provide an initial value directly in the graph data tab however. 
 50       The parameter is changed at runtime by using the graphmanager function I{graphmanager_set_graph_parameter_by_index}. 
 51       The indexing when using this function should be the same as the indexing that was used in yEd. 
 52       See the graphmanager documentation for further details. 
 53       Note that, except the image parameters, the supported nodes currently never has 2 parameters of the same type, so this 
 54       notation is currently unambiguous. Should there be a need to support multiple parameters 
 55       of the same type in a single node, the parser would need to be updated to handle this. 
 56       Note also that enums should use the specific type and not the generic wrapper name.\n 
 57       E.g. the API for vxConvertDepthNode specifies that the parameter "policy" 
 58       should be of type "vx_enum". However, as is specified in the parameter description, 
 59       "vx_enum" should be of the specific type "vx_convert_policy_e". The graph parser is designed to look 
 60       for the specific type, and not the generic "vx_enum" type. 
 61   
 62  Adding an image node to the graph 
 63  --------------------------------- 
 64   
 65    1. Image data nodes should be the flowchart object type "Process" 
 66    2. For image nodes that are input nodes or output nodes, we need to add (to the "Data" tab) 
 67       the node type and assign an array index in the corresponding image array, 
 68       e.g. [nodetype input_image[3]] for an input image that is accessed with input_images[3]. 
 69       Indexing should start from 0 and have no "gaps". 
 70       We also need to add the image format, e.g. [vx_df_image_e VX_DF_IMAGE_U8], to the data tab. 
 71       If no nodetype is added, the image node will be a virtual image, and the image type will be VX_DF_IMAGE_VIRT, 
 72       which means the OpenVX framework can decide by itself what image format it should have. 
 73       (A technical detail is that before the parser generates the C-code 
 74       it checks that all image formats are consistent throughout the graph, 
 75       and since it has the explicit information, it will actually never use the image format VX_DF_IMAGE_VIRT, 
 76       but instead use the image format it found that the OpenVX framework will have to use. 
 77       However, note that the image node type in this case is still a virtual image.) 
 78   
 79  See the documentation of the graph manager for information on how to set I/O images 
 80  when using the graph in algorithm code. 
 81   
 82  Connecting nodes in the graph 
 83  ----------------------------- 
 84   
 85    1. The edges (the lines between nodes) should have a target arrow, since we are working with directed graphs, 
 86       The type is arbitrary (one can choose the ones that give the best visual appearance, 
 87       e.g. quadratic, Bezier, polyline etc.) 
 88    2. If a function node has 2 input or 2 output edges, 
 89       they need to be labeled so that the parser can know 
 90       which edge corresponds to which parameter index in the OpenVX API. 
 91       To add a label to an edge, right-click on the edge and choose "Add label". 
 92       The 2 input edges should be labeled "in1" and "in2" respectively (note, no spacing or capital letters) 
 93       The 2 output edges should be labeled "out1" and "out2" respectively. 
 94       Note that for single input or output edges, no labeling is needed. 
 95   
 96  Userdata objects 
 97  ---------------- 
 98   
 99  This object should be of the type "User Message" (with an arrow-like right border). 
100  It should not be connected to any other object in the graph. 
101  Furthermore, it is an optional object, 
102  and there should be at most one such object per graph (or subgraph). 
103   
104  To set userdata parameters, right-click on the node and choose "Properties" and the "Data" tab. 
105  Enter any parameters that are global for the whole graph. 
106   
107  The value of such parameters can then be set by the user of the graph 
108  when the graph is registered to the graph manager, by passing in a UserData parameter object. 
109  For instance, if a reference width is needed to be specified, one can add the parameter 
110  [unsigned ref_width] to the UserData object. 
111   
112  See the documentation of the graph manager for further information on how to set the userdata parameter 
113  when registering the graph in the algorithm code. 
114   
115  Debug image nodes 
116  ----------------- 
117   
118  These nodes are primarily intended to be used during the design phase when a calculation graph 
119  is constructed and tested e.g. for bitexactness. It allows for the intermediate image data to be saved to 
120  external memory buffers that can be used for testing of the algorithm. They are typically removed before the 
121  production code is released. 
122   
123  Debug nodes should have the following information in the data tab: 
124   
125    1. An image debug node type should be specified, i.e. [nodetype debug_image[index]]. 
126       Note that it must be indexed, and indexing should start from 0 and have no "gaps". 
127    2. The debug node image format, e.g. [vx_df_image_e VX_DF_IMAGE_U8]. 
128       Since this is a node with an externally accessed memory buffer, it can not be of type VX_DF_IMAGE_VIRT. 
129   
130  See the documentation of the graph manager for information on how to set debug images 
131  when using the registered graph in algorithm code. 
132   
133  Subgraphs 
134  --------- 
135   
136  The graph manager object allows for "chaining" of graphs, where each graph drawn in yEd can be treated as a subgraph. 
137  This is useful for e.g. graphs with repetitive patterns. The chaining is done during registration time, 
138  Therefore nothing needs to be done in yEd in order to later use a particular graph as a subgraph. 
139   
140  See the graphmanager documentation for further details on how to use chaining of subgraphs. 
141
